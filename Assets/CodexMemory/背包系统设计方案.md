# 背包系统设计方案（背包 + 装备同屏 + 拖拽穿戴）

## 1. 目标
- 打开一个界面，同时看到：
  - 背包（Inventory）
  - 人物装备栏（Equipment）
- 支持拖拽交互：
  - 背包 -> 背包（交换/堆叠）
  - 背包 -> 装备栏（穿戴）
  - 装备栏 -> 背包（卸下）
  - 装备栏 -> 装备栏（换位/替换）
- 后续可扩展：属性联动、保存读档、物品来源系统（掉落/NPC/商店）

---

## 2. 总体架构
建议分 4 层：

1. 数据定义层（Definition）
- 负责物品静态信息（来自 SO）
- 不保存玩家当前数量

2. 运行时数据层（Runtime Model）
- 保存玩家当前背包和装备状态
- 例如：某格有几瓶药，武器位装备了什么

3. 规则服务层（Service）
- 负责所有可操作规则：能不能穿、能不能叠、能不能交换
- UI 不直接写规则

4. 表现与交互层（UI + Drag）
- 显示格子、图标、数量
- 捕获拖拽，最后调用 Service 进行真实数据修改

---

## 3. 核心数据结构（第一版）

### 3.1 ItemDef（ScriptableObject）
用途：物品静态定义

建议字段：
- `string itemId`
- `string itemName`
- `Sprite icon`
- `ItemType itemType`（消耗品/武器/护甲/材料...）
- `bool stackable`
- `int maxStack`
- `EquipSlotType equipSlotType`（仅装备类需要）

### 3.2 ItemStack（运行时）
用途：一份“物品 + 数量”

建议字段：
- `string itemId`
- `int count`

### 3.3 InventoryModel
用途：背包槽位数据

建议字段：
- `List<ItemStack> slots` 或固定长度 `ItemStack[] slots`
- `int capacity`

### 3.4 EquipmentModel
用途：装备槽位数据

建议字段：
- `Dictionary<EquipSlotType, ItemStack> equipped`

---

## 4. 规则服务（最关键）

### 4.1 InventoryService
职责：
- 新增物品（自动叠堆）
- 移除物品
- 背包内交换

### 4.2 EquipmentService
职责：
- 校验是否可装备（类型是否匹配槽位）
- 穿戴
- 卸下

### 4.3 ItemMoveService（统一入口，重点）
职责：
- 所有拖拽落点逻辑都走它
- 统一返回是否成功和失败原因

建议核心方法：
- `TryMove(SlotRef from, SlotRef to)`

其中 `SlotRef` 应能表示：
- 来源是背包哪一格
- 或来源是装备哪个槽位

典型分支：
1. 背包 -> 背包：交换/堆叠
2. 背包 -> 装备：校验后穿戴
3. 装备 -> 背包：尝试放入
4. 装备 -> 装备：同类槽位替换

---

## 5. UI 设计（同屏）

建议一个总面板：`BagEquipPanel`
- 左边：`InventoryGridView`
- 右边：`EquipmentPanelView`

### 5.1 SlotView（通用格子）
每个格子都复用同一个展示脚本，包含：
- 图标
- 数量文本
- 选中/高亮状态
- 空格子状态

### 5.2 DragGhost（拖拽跟随图标）
- BeginDrag：显示 ghost
- Drag：跟随鼠标
- EndDrag：隐藏 ghost

### 5.3 事件流
- UI 只产生事件（开始拖拽、结束拖拽）
- 不在 View 写业务规则
- Controller/Presenter 调用 `ItemMoveService.TryMove`
- 成功后刷新两侧 UI

## 6. 最小可落地迭代顺序（建议严格按顺序）

### 第一步：只做数据与规则（无 UI）
完成：
- `ItemDef`
- `ItemStack`
- `InventoryModel`
- `EquipmentModel`
- `ItemMoveService.TryMove`

验证方式：
- 先写简单测试代码，确认背包内交换/叠堆/穿戴规则正确

### 第二步：背包 UI 显示
完成：
- 背包格子渲染
- 图标与数量展示

### 第三步：背包内拖拽
完成：
- 背包 -> 背包
- 交换与堆叠

### 第四步：接装备栏
完成：
- 背包 -> 装备（穿戴）
- 装备 -> 背包（卸下）
- 校验错误提示（例如“该物品不能放入此槽位”）

### 第五步：装备属性联动
完成：
- 装备变化后通知角色属性系统重算

### 第六步：存档
完成：
- 保存背包+装备运行时数据
- 读档恢复

---

## 7. 与你当前项目的对接建议

你现在已有 MVC 风格 UI 和 Service 思路，建议这样接：
- `InventoryModel` / `EquipmentModel`：放数据
- `ItemMoveService`：放规则
- `BagEquipPanelController`：处理 UI 输入并调用服务
- `BagEquipPanelView`：只渲染，不做规则判断

保持原则：
- “规则只在 Service”
- “View 不直接改数据”
- “Controller 组织流程”
