三、Task 文件夹：每个脚本每个方法详细解释

TaskData.cs（任务定义）

成员作用在流程中的位置
taskId任务唯一标识（必须唯一）AcceptTask("2") 就是按这个匹配
taskName任务标题给左侧按钮和右侧详情显示
desc任务描述右侧详情描述文本
targetCount目标数量Runtime 初始化时赋值给 TargetCount
targetType进度事件类型NotifyTaskProgress 用它定位任务
rewardGold奖励值后续提交发奖使用

关键：taskId 不能重复、不能空。否则在 Service 初始化时会被跳过。

TaskStatus.cs（任务状态机）

状态语义谁触发切换
Locked未解锁后续前置任务链逻辑
Available可接取Initialize / runtime初建
InProgress进行中MarkAccepted()
Completed可提交ApplyProgress 达标后
Submitted已提交SubmitTask

TaskRunTime.cs（运行态）

这是“玩家当前任务进度”的真实存储，不是配置。

方法入参做了什么返回
ApplyProgress(int amount) 本次增量 1) 状态必须是 InProgress；
2) amount 必须 > 0；
3) 累加并不超过目标；
4) 达标后改为 Completed。 		无（内部改字段）
MarkAccepted() 无 把状态改到 InProgress 并记录时间。 无
MarkSubmitted() 无 把状态改到 Submitted。 无

TaskService.cs（核心业务层）

你现在的任务系统“是否正确运行”，几乎都由这个类决定。

字段解释

字段含义为什么要有
_definitionByTaskId定义字典按 id O(1) 查任务定义
_runtimeByTaskId运行态字典按 id O(1) 改状态/进度
_taskIdsByTargetType反向索引进度事件按类型快速命中任务
TasksChanged刷新事件通知 UI 重绘

方法解释（逐个）

方法什么时候调内部步骤关键注意
Initialize(definitions) GameMain.Start 清三张表 -> 遍历 definitions -> 校验 id -> 入定义字典；
你现在注释掉了“默认创建 runtime”，改成“动态接任务时创建”。 		会打印：definitions=?, runtime=?，目前 runtime 初始是 0。
AcceptTask(taskId) NPC对话/你的测试语句 1) 先查定义是否存在；
2) 若 runtime 不存在则创建；
3) 调 MarkAccepted()；
4) 触发 TasksChanged。 		taskId 必须和 SO 的 taskId 完全一致（包括字符串数字）。
SubmitTask(taskId) 点“提交”按钮 仅当状态是 Completed 才能提交，随后发刷新事件。 你当前 UI 的提交按钮可用状态来自 CanSubmit。
NotifyTaskProgress(targetType, amount) 击杀/采集等事件 通过 targetType 查到受影响任务 -> 逐个 ApplyProgress -> 有变化就发事件。 未接取的任务不在 runtime 字典，不会被推进。
BuildPanelItems() Controller 刷 UI 把 runtime+definition 映射成 TaskPanelItem 列表供 UI 使用。 只显示 runtime 里有的任务，即“已接取任务”。
TryGetRuntime 调试/详情查询 TryGet 封装。 只读查询。
SetTracked 追踪按钮 改 IsTracked 并刷新。 后续可接小地图引导。

TaskManager.cs（转发层）

方法作用在这条主线里的角色

成员/方法作用

方法作用

TaskPanelController.cs

方法作用关键点
SetTaskService注入业务层没有它就不会有任务数据
OnInitialize订阅 View / Model / Service 事件并初始化 UI会先 Model.SetTasks(service.BuildPanelItems())

CloseClicked关面板并恢复鼠标锁定逻辑标记设置 TaskManager.IsOpenTaskPanel=false

六、你现在这句 AcceptTask("2") 的实战检查清单

  1. SO检查：SecondTask.asset 的 taskId 必须正好是字符串 2。
  2. 注入检查：GameMain.Start 里已执行 TaskManager.Instance.SetTaskService(_taskService)。
  3. 绑定检查：UIManager.BindController(TaskPanel,...) 已执行。
  4. 显示检查：打开 TaskPanel 时左侧应出现“任务2”按钮。
  5. 日志检查：Initialize 打印 definitions=2, runtime=0；调用 Accept 后应触发 UI 刷新。

你的设计现在已经是“动态接任务显示”的正确方向：初始化不创建 runtime，接任务时才创建并显示。
