## 本次完成的核心功能

### 1. 背包持久化从“修改 SO”改为“JSON 存档”

之前背包主要依赖运行时修改 `PlayerBag.itemList` 和 `ItemSO.itemHeld`。这种方式在编辑器里看似可用，但在真正打包后不可靠，因为 `ScriptableObject` 资源本身不会被运行时永久写回。

本次改动后，`PlayerBag` 只作为运行时背包容器，真正持久化由 `BagSaveSystem` 写入：

```text
Application.persistentDataPath/bag_save.json
```

保存内容不保存 Unity 对象引用，只保存稳定数据：

```text
slotIndex
itemID
count
```

### 2. InventoryManager 成为背包唯一入口

`InventoryManager` 新增并统一负责：

- `TryAddItem(Item item, int count)`：拾取物品并保存。
- `SwapSlots(int fromSlot, int toSlot)`：拖拽交换槽位并保存。
- `SaveBag()`：把当前 `PlayerBag.itemList` 转成 JSON。
- `LoadBag()`：从 JSON 读出数据，再根据 `itemID` 还原成 `ItemSO`。
- `RegisterItemDefinition(Item item)`：注册物品定义，用于读档时通过 `itemID` 找回 `ItemSO`。
- `RegisterSceneItemDefinitions()`：启动时扫描场景中已有的拾取物，自动注册它们的 `ItemSO`。

启动顺序调整为：

```text
Awake()
-> 建立 InventoryManager 单例

Start()
-> RegisterSceneItemDefinitions()
-> LoadBag()
-> RefreshItem()
```

这个顺序解决了读档时 `itemDatabase` 尚未准备好，导致存档读到了但 UI 无法还原物品的问题。

### 3. 拾取物品接入统一存档流程

`ItemOnWord` 不再直接改 `playerInventory.itemList`，而是调用：

```text
InventoryManager.Instance.TryAddItem(item, addCount)
```

拾取成功后才销毁场景物体。这样拾取行为会同时完成：

- 放入 `PlayerBag.itemList`
- 刷新背包 UI
- 保存到 JSON

### 4. 拖拽交换接入统一存档流程

`ItemOnDrag` 不再直接持有 `myBag` 并手动交换列表，而是通过：

```text
InventoryManager.Instance.SwapSlots(currentSlotID, targetSlot.slotID)
```

这样拖拽交换后也会自动保存 JSON，并避免 UI 与数据来源不一致。

### 5. 修复拾取后卡死问题

原刷新逻辑中曾使用类似：

```csharp
while (childCount > 0)
{
    Destroy(child);
}
```

Unity 的 `Destroy` 会延迟到帧末执行，当前帧内 `childCount` 不会立刻减少，因此会死循环卡死。

本次改为固定次数倒序删除子物体：

```csharp
int childCount = instance.slotGrid.transform.childCount;
for (int i = childCount - 1; i >= 0; i--)
{
    Destroy(instance.slotGrid.transform.GetChild(i).gameObject);
}
```

### 6. 修复 SingleSlot 空引用

`SingleSlot.Awake()` 原来直接：

```csharp
slotButton = GetComponentInChildren<Button>();
slotButton.onClick.AddListener(ItemOnClicked);
```

当预制体里没有按钮或按钮未启用时会空引用。

本次增加了：

- 自动查找包含非激活子物体的按钮。
- 找不到按钮时输出警告，不再崩溃。
- `OnDestroy()` 移除监听。
- `itemInSlot`、`slotNum`、`slotImage` 的空值保护。

## 当前背包系统状态

现在背包逻辑是：

```text
ItemSO
-> 定义物品静态数据

InventoryManager.itemDatabase
-> itemID 到 ItemSO 的查询表

PlayerBag
-> 当前运行时背包槽位

bag_save.json
-> 真正跨重启保存的存档
```

## 后续使用规则

以后新增物品时：

1. 新建 `ItemSO`。
2. 填唯一且非空的 `itemID`。
3. 填名字、图标、描述、是否装备等静态信息。
4. 把它拖入 `InventoryManager.itemDatabase`。
5. 如果是地上拾取物，把它拖入对应 `ItemOnWord.item`。

## 仍需注意

- `PlayerBag.itemList` 需要提前有足够空槽。
- `ItemSO.itemID` 必须唯一。
- 场景扫描 `ItemOnWord` 只是兜底，不应替代完整的 `itemDatabase` 配置。
- 更长期的优化方向是做一个独立 `ItemDatabaseSO`，由它统一维护所有物品定义。
