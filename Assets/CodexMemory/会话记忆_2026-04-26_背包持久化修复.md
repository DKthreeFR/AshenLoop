# 会话记忆：背包持久化修复

日期：2026-04-26

## 本次讨论目标

当前背包系统在 Unity Editor 的部分模式下可以工作，但在 `Use Existing Build (Windows)` 或真正打包后，背包拾取和持久化表现不稳定。核心目标是：用最小改动让现有背包系统可以正常拾取、显示、拖拽交换，并在重启游戏后恢复上一次背包内容。

## 最终设计结论

`ScriptableObject` 不再承担持久化职责。打包后 `ItemSO` 和 `InventorySO` 是资源定义，不能依赖运行时修改它们来保存玩家背包。

现在各部分职责如下：

- `ItemSO`：物品定义，保存 `itemID`、名字、图标、描述等静态数据。
- `InventoryManager.itemDatabase`：物品定义库，用 `itemID` 找回对应的 `ItemSO`。
- `PlayerBag`：运行时背包容器，保存当前这一局内存中的格子列表。
- `bag_save.json`：真正持久化的数据，保存 `slotIndex`、`itemID`、`count`。
- `BagSaveSystem`：负责把 JSON 写入和读出 `Application.persistentDataPath`。

## 当前运行流程

游戏启动时：

```text
InventoryManager.Awake()
-> 建立单例

InventoryManager.Start()
-> RegisterSceneItemDefinitions()
-> LoadBag()
-> RefreshItem()
```

其中 `LoadBag()` 会读取 `bag_save.json`，然后根据存档里的 `itemID` 从 `itemDatabase` 找到对应 `ItemSO`，最后填回 `PlayerBag.itemList`。

拾取物品时：

```text
ItemOnWord.OnTriggerEnter()
-> InventoryManager.TryAddItem(item, addCount)
-> 修改 PlayerBag.itemList
-> InventoryManager.SaveBag()
-> BagSaveSystem.Save()
-> RefreshItem()
```

拖拽交换时：

```text
ItemOnDrag.OnEndDrag()
-> InventoryManager.SwapSlots(fromSlot, toSlot)
-> 交换 PlayerBag.itemList
-> SaveBag()
-> RefreshItem()
```

## 为什么还需要 PlayerBag

`PlayerBag` 仍然需要，但它现在不是最终存档，而是运行时背包容器。UI 和拖拽逻辑都需要一个当前可操作的格子列表，这个列表就是 `PlayerBag.itemList`。

真正跨重启保存的是 JSON：

```text
Application.persistentDataPath/bag_save.json
```

启动时会从 JSON 读出数据，再还原进 `PlayerBag.itemList`。

## 为什么需要 itemDatabase

JSON 不能稳定保存 Unity 对象引用，所以存档只保存稳定字符串 `itemID`。

例如：

```json
{
  "slots": [
    {
      "slotIndex": 0,
      "itemID": "RedBottle",
      "count": 1
    }
  ]
}
```

读档时系统只知道 `itemID = RedBottle`，还需要通过 `itemDatabase` 找到真正的 `RedBottle ItemSO`，才能拿到图标、描述、名字等显示数据。

因此以后新增物品时，至少要做：

1. 新建 `ItemSO`。
2. 填唯一且非空的 `itemID`。
3. 填图标、名字、描述等数据。
4. 把这个 `ItemSO` 拖到 `InventoryManager.itemDatabase`。
5. 如果它是地上可拾取物，再把它拖到对应 `ItemOnWord.item`。

## 为什么扫描场景里的 ItemOnWord

`InventoryManager.Start()` 会先扫描场景里的 `ItemOnWord`，把它们身上的 `ItemSO` 注册进 `itemDatabase`。

这是一个最小改动的兜底方案：你已有的地上物品本来就拖了 `ItemSO`，所以可以自动收集场景中可拾取物的定义。

但真正可靠的来源仍然应该是 `InventoryManager.itemDatabase`。如果某个物品只存在于存档里，但当前场景没有对应拾取物，那么只能靠 `itemDatabase` 找回它。

## 已修复的问题

1. 拾取物品时卡死。

原因是 `RefreshItem()` 里曾使用：

```csharp
while (childCount > 0)
{
    Destroy(child);
}
```

Unity 的 `Destroy` 会延迟到帧末执行，`childCount` 在同一轮循环中不会立刻减少，导致死循环。已改成固定次数倒序删除。

2. `SingleSlot.Awake()` 空引用。

原因是 `GetComponentInChildren<Button>()` 可能找不到按钮，但代码直接访问 `slotButton.onClick`。现在已增加空判断，并在找不到按钮时输出警告。

3. 重启后背包内容不恢复。

原因是读档时 `itemDatabase` 尚未准备好，或者没有对应 `ItemSO`。现在启动时先注册场景物品定义，再执行 `LoadBag()`；同时要求所有可能进入背包的物品都拖入 `InventoryManager.itemDatabase`。

## 当前注意事项

- `ItemSO.itemID` 必须唯一且不为空。
- `PlayerBag.itemList` 要提前有足够空槽，否则背包没有可放物品的位置。
- `InventoryManager.itemDatabase` 要包含所有可能被保存进背包的 `ItemSO`。
- 打包后不要期待 SO 文件本身被修改保存，持久化只看 JSON。
- 如果以后物品种类变多，建议升级为独立的 `ItemDatabaseSO`，不要长期依赖扫描场景物品。
